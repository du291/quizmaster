## BRACE Plan Snapshot

### B — Behavior
- Add dedicated mocked and backend WTR coverage for `Quiz.Edit.feature`.
- Prove editing an existing quiz title and description from the workspace updates the workspace list and the downstream quiz welcome page.
- Keep the slice bounded to edit-route prepopulation, submission, and post-save visibility; do not widen into quiz validations, question filtering, clipboard, stats, or unrelated workspace behaviors.

### R — Risks (top 1–5)
1. High / Slice-level behavior integrity: the edit flow depends on fetch, prepopulation, save, workspace navigation, and downstream quiz display, so shallow assertions could miss a real contract gap.
2. Medium / Scope / sequencing drift: quiz authoring has adjacent filter and validation behaviors that could easily bleed into this slice.
3. Medium / Harness / gate reliability: the full migration gate still inherits the host-aware backend wrapper baseline and low-grade legacy Playwright residue.
4. Medium / Environment / external variability: backend proof depends on the real quiz fetch and update API plus workspace refresh behavior.

### A — Assurances
- Reuse the existing `Quiz.CreateNew` seam, workspace quiz list entry point, and quiz backend helpers instead of widening production seams.
- Assert both the edited workspace list item and the edited quiz welcome-page description so the proof covers the whole edit round-trip.
- Keep targeted mocked and backend proof separate from the full gate and only claim closure if all three levels agree.
- Treat any contradiction between targeted runs and the full gate as a real signal, not as noise to ignore.
